Atomicity Definition
A requirement is atomic if it expresses ONE intentional system capability or behavior, regardless of implementation details, quality attributes, or operational conditions.

Atomicity Validation Rules
Single Intent Rule: The requirement must express one intentional system capability. Implementation steps, conditions, or constraints that serve this one capability do NOT violate atomicity.
Single Outcome Rule: The requirement must result in one verifiable outcome. Multiple paths to the same outcome (e.g., "authenticate via password or biometric") are still atomic.
No Unrelated Conjunction Rule: The requirement must not combine UNRELATED or INDEPENDENT capabilities. Conjunctions are acceptable when they describe:
  - Steps of a single capability (e.g., "validate and store")
  - Variants of the same action (e.g., "upload or download")
  - Conditions serving one intent (e.g., "log in on web or mobile")
No Embedded Obligations Rule: The requirement must not embed secondary, independent obligations. Quality constraints (security, performance) that modify HOW the capability works are NOT embedded obligations.

Examples of Atomic Requirements:
"The system shall encrypt data during transmission" → Atomic: one capability (secure transmission), encryption is HOW not a separate obligation
"The system shall validate and store user input" → Atomic: validation+storage serve one capability (input processing)
"Users shall log in via password or biometric" → Atomic: one capability (authentication) with multiple paths
"The system shall support Windows, Mac, and Linux" → Atomic: one capability (cross-platform support) with enumerated platforms

Examples of Non-Atomic Requirements:
"The system shall authenticate users AND generate reports" → Non-atomic: two independent capabilities
"The system shall validate input and send email notifications" → Non-atomic: validation and notification are unrelated
"The user shall create an account and the system shall backup data" → Non-atomic: two separate obligations

Mandatory Interpretation Rules - Do NOT Flag These as Non-Atomic:
Enumerations of platforms, environments, technologies, or conditions SHALL be treated as serving one unified capability.
Bidirectional verbs (e.g., upload/download, send/receive, read/write) SHALL be treated as one capability when applied to the same object and interface.
Non-functional constraints (security, performance, availability, reliability, compliance, scalability) SHALL NOT be interpreted as additional intents or embedded obligations.
Specifying HOW a capability is performed (e.g., "encrypt using AES", "respond within 2 seconds", "via REST API") does NOT introduce a second intent.
Sequential steps that serve one unified capability SHALL be treated as atomic (e.g., "authenticate then authorize" is one capability: secure access).
