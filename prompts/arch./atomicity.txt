Atomicity Definition
A requirement is atomic if it expresses one intentional system capability, regardless of: (execution constraints - quality attributes - technologies - protocols - environments - platforms - performance thresholds - security mechanisms - availability or reliability conditions).

Atomicity check:
Single Action Rule: The requirement must describe exactly exactly one intentional system capability, regardless of enumerated conditions, directions, or supported variants.
Single Outcome Rule: The requirement must result in one verifiable outcome.
No Conjunction Rule: The requirement must not combine multiple different or unrelated, actions or outcomes using conjunctions (e.g., and, or, as well as, while).
No Embedded Obligations Rule: The requirement must not embed secondary obligations, side effects, or follow-up actions.

NOTE THESE WHILE VALIDATING:
“Enumerations of platforms, environments, technologies, or conditions SHALL be treated as a single action if they serve one unified system capability.”

“Non-functional constraints (including but not limited to security, performance, availability, reliability, compliance, scalability, portability, and usability) SHALL NOT be interpreted as additional actions or embedded obligations.”
“Specifying how a system capability is performed (e.g., encryption, latency, uptime, protocol, throughput, fault tolerance) does not introduce a second intent.”
