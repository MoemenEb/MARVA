Completion Definition
A requirement is complete if all essential elements necessary to understand the intended system capability are present.

Essential Elements:
1. ACTOR: Who/what performs or is affected (may be implicit, e.g., "the system")
2. ACTION: The intended system behavior or capability that must be provided
3. OBJECT: The object or target of the action (what is acted upon or produced)
4. CONDITION: Conditions or constraints (only if essential to define when/where the requirement applies)
5. FLOW: How an action is going to be done, or at least the outline of the operation.

Completion Validation Rules
Actor Rule: An actor must be identifiable (explicit, implicit, or inferable). Actors may be users, systems, documents, data, or artifacts.
Action Rule: The intended system behavior or capability must be clearly stated. Both behavioral actions (e.g., "validate") and declarative intents (e.g., "comply with GDPR") are acceptable.
Object Rule: The object or target of the action must be identifiable. What is being acted upon or produced must be clear.
Condition Rule: Conditions must be present only if they are essential to understanding when/where the requirement applies. If the requirement applies universally, conditions are not needed.

Examples of Complete Requirements:
"The system shall log all transactions" → Actor: system (implicit), Action: log, Object: transactions, FLOW: log is a well known flow.
"Users must authenticate before accessing data" → Actor: users, Action: authenticate, Object: implicit (system/data), Condition: before accessing data
"Must comply with GDPR" → Actor: system (implicit), Action: comply, Object: GDPR (even though non-behavioral, intent is clear)

Examples of Incomplete Requirements:
"Authentication is required" → Actor unclear (system? users? both?)
"Shall handle errors" → Object missing (which errors? how handle?)
"The system shall send notifications" → Condition missing (when? to whom? under what circumstances?)

Mandatory Interpretation Rules - Do NOT Flag These as Incomplete:
Missing quality metrics or thresholds (e.g., "fast response" without defining milliseconds) SHALL NOT be treated as incompleteness.
Missing implementation details (e.g., "how" the system achieves the behavior) SHALL NOT be treated as incompleteness.
Non-behavioral or declarative requirements (e.g., "comply with standard X") SHALL be considered complete if the intent is fully stated.
Environmental, legal, or organizational context (e.g., deployment platform, regulations) SHALL NOT be assumed missing unless explicitly required for the core capability.
Lack of specificity in scope SHALL NOT be treated as incompleteness if the requirement's essential elements are present.
